# fallow on web-forth (companion to `fallow:fallow`)

The official **`fallow:fallow`** skill is the general tool manual — every command,
all the issue types, the flags, the MCP tools, exit codes, config schema. Use it for
generic usage. **This skill does not repeat that.** It holds only what is
web-forth-specific: how fallow is wired into this repo's commit gate, the per-task
order, this repo's config, and the tradeoffs already settled. Read both; they compose.

`fallow` is the static-analysis gate for this repo: dead code (unused
files/exports/deps/members), duplication, cyclomatic/cognitive complexity, and
architecture boundaries. A Claude Code **PreToolUse hook**
(`.claude/hooks/fallow-gate.sh`) runs it on every `git commit`/`git push` and
**blocks the command (exit 2) when the audit returns `verdict:"fail"`.** So fallow is
not advisory here. `--no-verify` is forbidden (a failing audit means fix the code).

fallow is installed global (v2.86.0+). The hook was installed via
`fallow hooks install --target agent` (writes `.claude/settings.json` +
`.claude/hooks/fallow-gate.sh`, plus a managed block in `AGENTS.md`). It runs via
Git Bash + jq on Windows and **fails OPEN** on missing jq/fallow or a runtime error;
only a real `verdict:fail` is a hard block.

## When to run it (the per-task order)

**build/simplify → `fallow audit` → advisor → commit.** fallow runs BEFORE the
pre-commit advisor so the advisor reviews already-clean code, not dead-code noise.

- `fallow audit --format json --quiet --explain` — the exact command the hook runs.
  Reproduce it to predict the gate outcome. Read top-level `verdict`; the human report
  also lists health findings (complexity, unit size) that do NOT flip the verdict.
- `fallow dead-code` — full dead-code scan.
- `fallow health` — complexity / maintainability score (advisory; exits non-zero on a
  low score but is NOT the commit gate — only `audit`'s `verdict` gates).
- `fallow dupes` — duplication.
- `fallow fix` — auto-removes unused exports/deps. Prefer deleting/de-exporting by
  hand; only `fix` code you are sure is truly dead.

The gate defaults to `gate=new-only`: only findings the current changeset INTRODUCES
affect the verdict; inherited findings on touched files are reported with
`introduced:false` and do not block.

## `.fallowrc.json` (repo root)

- `entry`: `["src/index.{ts,...}", "src/main.{ts,...}", "src/entry.{ts,...}"]` —
  per-package reachability roots. engine + cli root at `src/index.ts`; client boots
  at `src/entry.ts` (with `src/main.ts` also present). Keep all three globs: a wrong
  entry glob loses a package's import graph and false-flags a real dependency.
- `workspaces.packages: ["packages/*"]`.
- `ignorePatterns`: `repos/**` (vendored effect/foldkit/codemirror, read-only),
  `**/*.d.ts`, `**/prelude.generated.ts` (a build artifact regenerated by the engine's
  pretest/pretypecheck script — not hand-authored source).
- `publicPackages: ["@web-forth/engine"]` ONLY. The engine is the library (its exports
  are consumed by cli + client). client and cli are leaf apps — marking them public
  makes dead-export detection vacuous (nothing imports a leaf app).
- `ignoreDependencies: []`, `ignoreExports: []` — kept empty on purpose. The client is
  Foldkit/TEA, so if fallow later flags the canonical duplicate-export pattern
  (`Message`/`Model`/`init` recurring across page submodels) or a barrel-only type,
  suppress by EXACT export name (never `["*"]`), and only after confirming a real
  importer exists (grep direct + the barrel dir). Delete genuinely-dead exports; do not
  suppress them.
- `duplicates.minOccurrences: 3`, ignoring `repos/**`.

## The two things this repo actually hit

### 1. Splitting a big install fn into files can clone the `def` wrapper
When `packages/engine/src/forth.ts` was split into `primitives/*.ts`, each install fn
opened with the same `const def = (name, routine, immediate=false) => f.definePrimitive(...)`
closure. Four identical copies tripped fallow `dupes` (a 9-line clone group). Fix:
one shared `primitives/shared.ts` exporting `makeDef(f)`; each install fn is
`const def = makeDef(f)`. **Lesson:** when extracting sibling modules that share a tiny
boilerplate closure, hoist it to a shared helper in the same commit — do not copy it.

### 2. Grouped-install fns are long BY DESIGN (accepted unit-size deduction)
`installCore` / `installControlFlow` / `installExtended` each register dozens of
primitives in one fn, so they exceed fallow's 60-line unit-size threshold. This is
health `unit size`, NOT a `verdict:fail`, and NOT complexity — avg cyclomatic is 1.3
(flat `def(...)` registration boilerplate). We deliberately do NOT fragment them into
`installStack`/`installArith`/... : that re-introduces the exact sprawl the split
removed, for zero real complexity. `fallow health` sitting at ~79/B from unit-size +
historical churn (hotspots) is expected and accepted. The commit gate (`audit`
verdict) is what must stay green.

Actual complexity (cyclomatic > 20 AND cognitive > 15, or CRAP > 30) IS worth fixing —
extract named helpers (as `token` in `forthLanguage.ts` and `runTokens` in `forth.ts`
were). Keep extracted engine helpers throwing (no try/catch) so `THROW`/`EVALUATE`
propagation (§V.17/§V.18) is preserved.
